# Aurora — Product Vision

## Concept

Aurora is an agentic RPG where the player defines the universe and the LLM agent builds, inhabits, and narrates it in real time. There is no fixed story — each session is unique, fully generated by the agent from the player’s initial setting. NPCs have their own personality, memory, and voice. The world reacts visually with images, video, music, and sound effects generated in real time as the narrative unfolds.

---

## Product Pillars

| Pillar | Description |
|---|---|
| **Generative narrative** | The player defines the setting; the agent builds the world, story, and characters |
| **Deep NPCs** | Personality, persistent memory, and unique voice per character |
| **Real-time media** | Images, video, music, SFX, and voice generated automatically by the agent |
| **Reactive interface** | WebGL/R3F frontend that responds dynamically to game events |
| **Multimodal** | Text and voice interaction interchangeable |

## LLM Agent Layer

Aurora uses **Mastra** (TypeScript) as the agent framework, running in the same Bun process as the API Gateway. A **single agent with multiple tools/skills** — not multiple specialized LLMs. The agent takes on different roles through dynamic system prompts and context switching.

### Agent Roles

| Role | Responsibility |
|---|---|
| **Narrator** | Builds the story, describes scenes, manages narrative flow |
| **NPC** | Embodies characters with individual personality and memory |
| **Art Director** | Decides what media to generate, when, and with what parameters |
| **Game Master** | Evaluates player actions, applies world rules, resolves conflicts |

### Agent Tools

#### Media generation (real time)
| Tool | Description |
|---|---|
| `generate_image` | Generates scene or character image — only if no suitable asset exists |
| `generate_video` | Generates cinematic or short animation — only if no suitable asset exists |
| `generate_music` | Generates adaptive soundtrack to scene mood |
| `generate_sfx` | Generates contextual sound effect |
| `generate_voice` | Generates voice for the active NPC (Qwen TTS) |

#### Asset library
| Tool | Description |
|---|---|
| `generate_asset` | Generates and catalogs an asset of any type (image, video, music, SFX, voice) with metadata (type, emotion, scene, character, tags) |
| `search_assets` | Searches assets by semantic metadata and type — “angry NPC”, “tense battle music”, “rain SFX” |
| `compose_scene` | Combines existing assets to build a scene without generating again |

#### Game state
| Tool | Description |
|---|---|
| `update_game_state` | Updates game state (inventory, quests, map) |
| `recall_memory` | Retrieves NPC or past event memory (MCP) |
| `trigger_ui_event` | Triggers visual effect on the frontend (explosion, zoom, etc.) |

### Media decision flow

The agent follows this flow before generating any visual or audio asset:

```
Does a suitable asset exist in the library?
  ├── Yes → search_assets → compose_scene / use directly
  └── No → generate_* → generate_asset (catalog for future use)
```

This pattern applies to all media types:

| Type | Reuse example |
|---|---|
| Image | NPC angry portrait generated once, reused in each confrontation |
| Video | Generic battle cinematic adapted with compositing |
| Music | Tension track cataloged, reused in similar scenes |
| SFX | Footsteps, rain, fire — generated once per environment |
| Voice | Frequent dialogue lines pre-generated per NPC |

### LLM compatibility

The agent is compatible with any OpenAI-compatible or Anthropic-compatible provider — the underlying model is interchangeable without changing agent logic.

## Media Layer (comfy-diffusion)

All media generation runs on **comfy-diffusion** as a Python library, without a ComfyUI server or node system.

| Media Type | Model | comfy-diffusion module |
|---|---|---|
| Images (characters, scenes) | Flux / SDXL | `sampling`, `conditioning`, `vae` |
| Video (cinematics, actions) | WAN 2.1 / LTXV | `conditioning.wan_*`, `conditioning.ltxv_*` |
| Adaptive music | ACE Step 1.5 | `audio.encode_ace_step_15_audio` |
| Sound effects | ACE Step 1.5 | `audio` |
| NPC voice | Qwen TTS | `audio` (future) |

### Character visual consistency

The system uses two distinct modes depending on context:

**Onboarding mode — Character creation**
SDXL + IP-Adapter generates the character’s canonical reference image. The player describes or customizes their character; the base image is generated and persisted in the Asset Store as a permanent reference.

| Step | Description |
|---|---|
| 1. Player input | Text description + optional references |
| 2. Base generation | SDXL generates high-quality image |
| 3. Refinement | IP-Adapter adjusts visual identity |
| 4. Persistence | Canonical image saved in Asset Store as character reference |

**Gameplay mode — Use in session**
Flux Kontext / Qwen Edit receive the canonical reference image and generate new scenes, expressions, and situations while keeping character identity automatically. No LoRAs, no retraining.

| Strategy | Use | Model |
|---|---|---|
| **Edit model** (primary) | All gameplay scenes with character present | Flux Kontext, Qwen Edit |
| **LoRA** (fallback) | Highly stylized characters or when edit model unavailable | LoRA generated at onboarding |

Gameplay flow: agent retrieves reference image from Asset Store → passes image + new scene prompt to edit model → resulting image keeps character identity.

## Frontend (R3F + WebGL)

The interface is reactive and dynamic — not a static chat with attached images. The canvas responds to game events in real time.

### Main components

| Component | Description |
|---|---|
| **Chat / Voice** | `assistant-ui` — text/voice conversation interface |
| **Scene Viewer** | R3F canvas showing the current scene image/video |
| **Effects Layer** | Reactive WebGL effects (magic particles, explosions, fog, rain) |
| **HUD** | Inventory, character stats, active quests — updates with game state |
| **NPC Portrait** | Animated portrait of the active NPC with basic lip-sync |

### UI events triggered by the agent

| Event | Visual effect |
|---|---|
| `combat_start` | Screen shake + combat music |
| `spell_cast` | Magic particles on the canvas |
| `explosion` | Flash + particles + SFX |
| `scene_change` | Crossfade + new scene image |
| `cinematic` | Fullscreen video |
| `ambient_change` | Music change + ambient effects |

## Development Phases (tentative)

### Phase 1 — Core Loop
Narrator agent + text chat + scene image generation. No voice, no video, no effects. Basic loop: player describes setting → agent builds world → narrative with images.

### Phase 2 — Deep NPCs ⭐ MVP
Persistent NPC personality and memory. Multiple characters in scene. Basic inventory and quest system.

### Phase 3 — Full Media
Cinematic video at key moments. Adaptive music. Contextual SFX. NPC voice with Qwen TTS.

### Phase 4 — Reactive Frontend
R3F interface with WebGL effects. Dynamic HUD. Navigable map. Visual combat system.

### Phase 5 — Player Voice
Voice input interchangeable with text. Basic lip-sync on NPC portraits.

### Phase 6 — Polish and Scalability
Character visual consistency via LoRA. Latency optimization. Multi-session. Polished onboarding.

## Key Architectural Decisions

- **Single agent, not multi-agent** — operational simplicity, single conversation context, less coordination latency
- **Media Service separate from Agent Service** — the agent doesn’t block waiting for generation; media requests go to a queue and the result arrives via WebSocket
- **comfy-diffusion as library, not ComfyUI server** — full pipeline control, no node system overhead, testable in CI
- **ElysiaJS as gateway, not app server** — Bun handles WebSockets and streaming efficiently; heavy logic lives in Python
- **Interchangeable LLM** — agent logic is not coupled to any specific provider
