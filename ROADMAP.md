# Aurora — Development Roadmap

> **Methodology:** [nerds-vibecoding-survivor-toolkit](https://github.com/quinteroac/nerds-vibecoding-survivor-toolkit)
> **Per-iteration flow:** `define requirement` → `approve req` → `create prototype` → next iteration
> **Testing:** TDD during prototyping + manual player review
> **Context:** Update `PROJECT_CONTEXT` at the end of each phase

---

## Conventions

- Each **iteration** produces a concrete, functional deliverable
- The cycle is always: **Define → Approve → Prototype**
- Do not advance to the next iteration without approving the current requirement
- `[📋 Context Update]` marks a PROJECT_CONTEXT update point

---

## Phase 1 — Core Loop

> **Goal:** Functional basic loop — player describes a setting, the agent builds the world and narrates with real-time generated images.

---

### It.01 — Project Scaffold & Monorepo

**Cycle:** define requirement → approve req → create prototype

**Goal:** Base project structure with all services scaffolded and communicating.

**Deliverables:**
- Monorepo with workspaces: `apps/backend` (Bun + ElysiaJS), `apps/frontend` (React), `services/media` (Python + FastAPI)
- `bun nvst init` run — `.agents/` and `docs/nvst-flow/` structure in place
- Health check endpoints in backend and media service responding
- Environment variables and base configuration (`.env.example`)
- Bun scripts to run the full stack (backend, frontend, media service) and automation tasks

---

### It.02 — Media Service Base

**Cycle:** define requirement → approve req → create prototype

**Goal:** FastAPI wrapper over comfy-diffusion with a working image generation endpoint.

**Deliverables:**
- `POST /generate/image` — accepts prompt, returns base64 image or URL
- Integration with comfy-diffusion (Flux / SDXL basic pipeline)
- Error and timeout handling
- `GET /health` endpoint with pipeline status

---

### It.03 — Mastra Agent Base

**Cycle:** define requirement → approve req → create prototype

**Goal:** Mastra agent running on ElysiaJS with Narrator role and integrated `generate_image` tool.

**Deliverables:**
- Mastra agent with Narrator system prompt
- Tool `generate_image` → calls Media Service via HTTP
- Endpoint `POST /chat` — accepts player message, returns narrative response
- WebSocket `/ws` — agent response streaming to client

---

### It.04 — Chat Interface

**Cycle:** define requirement → approve req → create prototype

**Goal:** Frontend with `assistant-ui` connected to the backend via WebSocket showing narrative in real time.

**Deliverables:**
- Chat interface with `assistant-ui`
- WebSocket connection to backend
- Streaming response display
- Display area for scene image generated by the agent
- Initial setting: input where the player describes the game universe

---

### It.05 — Game Session & World Building

**Cycle:** define requirement → approve req → create prototype

**Goal:** Full game loop — from initial setting to continuous narrative with scene images.

**Deliverables:**
- Onboarding flow: player enters setting → agent builds and describes the world
- Agent decides when to call `generate_image` (basic Art Director)
- Scene image shown on frontend when it arrives
- Session state in memory (no persistence yet)
- Narrator prompt engineering: narrative coherence, evocative descriptions

## Phase 2 — Deep NPCs ⭐ MVP

> **Goal:** NPCs with personality and persistent memory. Basic inventory and quests. This is the playable MVP.

---

### It.06 — Persistence Layer

**Cycle:** define requirement → approve req → create prototype

**Goal:** Operational database with schema for game state, NPC memory, and asset store.

**Deliverables:**
- PostgreSQL + pgvector + Redis running via Podman
- Initial schema: `sessions`, `npcs`, `npc_memory`, `game_state`, `assets`
- Migrations with chosen tool (Drizzle ORM or Kysely)
- Redis for session cache and Media Service job queue
- Agent tool `update_game_state` operational

---

### It.07 — NPC Personality Engine

**Cycle:** define requirement → approve req → create prototype

**Goal:** The agent can embody NPCs with defined, consistent personality across messages.

**Deliverables:**
- NPC schema: name, background, motivations, quirks, voice tone
- Dynamic system prompt that injects the active NPC’s personality
- Context switching: agent alternates between Narrator and NPC roles fluidly
- NPC portrait generated with SDXL on first appearance → persisted in Asset Store
- At least 3 test NPC archetypes (ally, neutral, antagonist)

---

### It.08 — NPC Memory System

**Cycle:** define requirement → approve req → create prototype

**Goal:** NPCs remember what happened in previous sessions and react accordingly.

**Deliverables:**
- Tool `recall_memory` — semantic search over interaction history (pgvector)
- Tool `store_memory` — agent automatically stores relevant events
- Memory embeddings on session close
- NPCs reference past events naturally in dialogue
- Mastra MCP integrated for memory management

---

### It.09 — Inventory & Quest System

**Cycle:** define requirement → approve req → create prototype

**Goal:** Inventory system and items generated by the agent, with dynamic quests.

**Deliverables:**
- Extended tool `update_game_state`: inventory, active quests, completed quests
- Agent generates items with name, description, and image (via `generate_image`)
- Quests created dynamically by the agent from the narrative
- Game state queryable by the agent each turn (session context)

---

### It.10 — Asset Store & Reuse

**Cycle:** define requirement → approve req → create prototype

**Goal:** The agent reuses already-generated assets instead of always regenerating.

**Deliverables:**
- Tool `search_assets` — semantic search by type, emotion, character, tags (pgvector)
- Tool `generate_asset` — generates and catalogs any asset type with metadata
- Decision flow: search before generating
- Basic metrics: reuse vs new generation ratio

---

## Phase 3 — Full Media

> **Goal:** Full multimedia stack — adaptive music, contextual SFX, cinematic video, and NPC voice.

---

### It.11 — Adaptive Music

**Cycle:** define requirement → approve req → create prototype

**Goal:** Agent generates and changes background music according to scene mood.

**Deliverables:**
- Tool `generate_music` — ACE Step 1.5 generates 30s track from mood/context
- Agent calls `generate_music` on scene changes, combat, dramatic moments
- Audio plays continuously on frontend with fade between tracks
- Tracks cataloged in Asset Store and reused for similar mood

---

### It.12 — Contextual SFX

**Cycle:** define requirement → approve req → create prototype

**Goal:** Sound effects generated automatically for actions and environments.

**Deliverables:**
- Tool `generate_sfx` — ACE Step 1.5 for short effects (< 5s)
- Agent generates SFX for: footsteps, magic, combat, ambience (rain, fire, wind)
- SFX play on frontend in sync with narrative
- Reusable SFX library in Asset Store

---

### It.13 — Cinematic Video

**Cycle:** define requirement → approve req → create prototype

**Goal:** 3–5 second cinematics at key narrative moments (major battles, plot twists).

**Deliverables:**
- Tool `generate_video` — WAN 2.2 / LTXV pipeline in Media Service
- Agent uses `generate_video` selectively (not every turn — only special moments)
- Frontend shows fullscreen video with smooth transition
- “Narrative wait” UX — player knows something special is being generated

---

### It.14 — NPC Voice

**Cycle:** define requirement → approve req → create prototype

**Goal:** Each NPC has a unique voice generated by Qwen TTS when speaking.

**Deliverables:**
- Tool `generate_voice` — Qwen TTS with per-NPC voice parameters (pitch, speed, accent)
- Unique voice defined in NPC schema and persisted
- Low-latency audio streaming to client
- Frequent lines per NPC pre-generated and cached

---

## Phase 4 — Reactive Frontend

> **Goal:** WebGL/R3F interface that responds dynamically to game events. Aurora’s visual differentiator.

---

### It.15 — R3F Scene Viewer

**Cycle:** define requirement → approve req → create prototype

**Goal:** Three.js canvas showing the current scene image with smooth transitions.

**Deliverables:**
- R3F canvas that renders the current scene image
- `scene_change` transition: smooth crossfade between images
- Layout: chat on left panel, canvas on right panel
- Preparation of UI event architecture (frontend event bus)

---

### It.16 — WebGL Effects Layer

**Cycle:** define requirement → approve req → create prototype

**Goal:** Reactive visual effects over the canvas that the agent can trigger.

**Deliverables:**
- Particle system for: magic, explosions, fog, rain, fire
- Agent tool `trigger_ui_event` → WebSocket → visual effect on frontend
- Implemented events: `spell_cast`, `explosion`, `combat_start`, `ambient_change`
- Basic GLSL shaders for screen effects (flash, shake, vignette)

---

### It.17 — Dynamic HUD

**Cycle:** define requirement → approve req → create prototype

**Goal:** HUD with inventory, stats, and quests updating in real time with game state.

**Deliverables:**
- Inventory panel: items with thumbnail image generated by the agent
- Active quests panel with narrative progress
- Basic character stats (health, energy, etc. as defined by the game world)
- HUD updates automatically when the agent calls `update_game_state`

---

### It.18 — NPC Portrait & Animation

**Cycle:** define requirement → approve req → create prototype

**Goal:** Animated portrait of the active NPC with dynamic expression by emotional context.

**Deliverables:**
- NPC Portrait component: active NPC portrait image in canvas corner
- Expressions: agent specifies emotion (neutral, angry, happy, scared)
- Basic lip-sync during TTS audio playback
- Smooth transition between NPCs when changing active character

---

### 📋 Context Update — Post Phase 4

## Phase 5 — Player Voice

> **Goal:** Voice input interchangeable with text.

---

### It.19 — Voice Input

**Cycle:** define requirement → approve req → create prototype

**Goal:** Player can speak instead of typing; transcription reaches the agent.

**Deliverables:**
- Speech-to-text on frontend (Web Speech API or local Whisper)
- Voice/text toggle in `assistant-ui`
- Real-time transcription visible while the player speaks
- Silence handling and end-of-turn detection

---

## Phase 6 — Polish

> **Goal:** Visual consistency, latency optimization, and polished onboarding.

---

### It.20 — Character Onboarding

**Cycle:** define requirement → approve req → create prototype

**Goal:** Main character creation flow with generated and persisted canonical image.

**Deliverables:**
- Onboarding UI: player describes their character (text + optional references)
- SDXL + IP-Adapter generates high-quality canonical image
- Canonical image persisted in Asset Store as permanent reference
- Agent uses the reference in all scenes where the character appears

---

### It.21 — Character Consistency (Flux Kontext)

**Cycle:** define requirement → approve req → create prototype

**Goal:** Main character images during gameplay keep consistent visual identity.

**Deliverables:**
- Flux Kontext pipeline in Media Service: receives canonical image + scene prompt
- Tool `generate_image` updated to pass reference image when character is in scene
- LoRA as fallback for highly stylized characters
- Subjective visual evaluation: “clearly the same character”

---

### It.22 — Latency & Performance

**Cycle:** define requirement → approve req → create prototype

**Goal:** Reduce perceived response times and improve experience fluidity.

**Deliverables:**
- Speculative pre-generation of likely assets (next scene)
- Priority queue: voice > image > music > video
- Narrative loading indicators (agent “describes” while generating)
- Latency metrics per asset type logged

---
