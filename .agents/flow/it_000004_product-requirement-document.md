# Requirement: Chat Interface

## Context

Aurora needs a playable front-end to close the core loop defined in Phase 1.
Iterations 01–03 delivered the monorepo scaffold, media service, and Mastra agent backend.
Iteration 04 adds the browser UI: an onboarding screen where the player describes their
universe and a live chat powered by `assistant-ui` that streams the agent's narrative and
renders scene images as they arrive from the backend WebSocket.

## Goals

- Give the player a first-run onboarding flow to define their game universe before entering the chat.
- Connect `assistant-ui` to the backend WebSocket so narrative responses stream in real time.
- Display scene images (sent by the agent via the WebSocket) in a dedicated area alongside the chat.
- Adhere strictly to all project standards: TypeScript strict mode, functional React components, ESLint + Prettier, no `console.log` in production paths, Vitest for unit tests.

## User Stories

### US-001: Universe Onboarding Screen

**As a** player, **I want** to be greeted by a full-screen onboarding form where I describe my game
universe **so that** the agent can build and narrate a world tailored to my setting before the chat begins.

**Acceptance Criteria:**
- [ ] A dedicated `/` route (or initial app state) shows a full-screen onboarding view with a textarea and a "Begin Adventure" submit button.
- [ ] The textarea is focused on mount and accepts multi-line input.
- [ ] Submitting with an empty textarea is blocked; a visible validation message is shown.
- [ ] On submit, the setting text is sent to the backend as the first player message, and the app transitions to the chat view.
- [ ] Typecheck (`tsc --noEmit`) and lint (`eslint`) pass with no errors.
- [ ] Visually verified in browser: onboarding screen renders, validation fires, submission navigates to chat.

---

### US-002: Streamed Narrative Chat

**As a** player, **I want** to type messages in a chat interface and see the agent's narrative
response appear word-by-word as it streams **so that** the game feels alive and responsive.

> **Implementation Note:** Before implementing the `WebSocketRuntime`, fetch and read
> https://www.assistant-ui.com/docs to confirm the correct `useAssistantRuntime` interface
> and custom runtime contract. Do not infer the API from training data — `assistant-ui`
> evolves frequently.

**Acceptance Criteria:**
- [ ] `assistant-ui` chat component is mounted on the chat view (`/chat` route or app state).
- [ ] Player messages are sent to the backend over WebSocket.
- [ ] Agent responses stream token-by-token into the message bubble using `assistant-ui`'s streaming API.
- [ ] A typing/loading indicator is visible while the agent is generating.
- [ ] Completed messages are retained in the conversation history for the session.
- [ ] Typecheck and lint pass with no errors.
- [ ] Visually verified in browser: message sends, streaming text appears incrementally, full message persists.

---

### US-003: Scene Image Display

**As a** player, **I want** a dedicated area next to the chat that shows the latest scene image
generated by the agent **so that** I can visualize the world being narrated.

**Acceptance Criteria:**
- [ ] A scene image panel (right panel on desktop layout) is rendered alongside the chat.
- [ ] When the backend WebSocket sends a `scene_image` event (base64 or URL), the panel updates with the new image.
- [ ] While no image has arrived yet, the panel shows a placeholder (dark background with a subtle "Awaiting scene…" label).
- [ ] Image transitions use a CSS crossfade (opacity transition) between the old and new image.
- [ ] The panel is responsive: on narrow viewports (< 768 px) the image collapses below the chat.
- [ ] Typecheck and lint pass with no errors.
- [ ] Visually verified in browser: placeholder renders on load, new image replaces placeholder with crossfade, responsive layout is correct.

---

### US-004: WebSocket Connection Status Indicator

**As a** player, **I want** to see a visible indicator of the WebSocket connection state **so that**
I know when the game is connected and when it is not.

**Acceptance Criteria:**
- [ ] A small status badge (e.g. green dot = connected, red dot = disconnected) is always visible in the UI.
- [ ] When the WebSocket closes unexpectedly, the badge updates and a non-blocking toast/banner informs the player.
- [ ] The client attempts a single automatic reconnect after a 3-second delay on unexpected disconnect.
- [ ] Typecheck and lint pass with no errors.
- [ ] Visually verified in browser: disconnect simulation (DevTools → offline) shows the indicator change and reconnect attempt.

> **Note:** US-004 is part of the iteration scope but is not required for the primary "done" definition.
> The core done criteria is satisfied by US-001 + US-002 + US-003 working end-to-end.

## Functional Requirements

- **FR-1:** The frontend is a Vite + React 18 + TypeScript SPA living at `apps/frontend`, following all conventions in `PROJECT_CONTEXT.md`.
- **FR-2:** `assistant-ui` is used as the chat component via a custom `WebSocketRuntime` built with `useAssistantRuntime`; no REST adapter and no custom chat bubble implementation from scratch. Before implementing, fetch and read https://www.assistant-ui.com/docs to verify the current API contract — do not infer it from training data.
- **FR-3:** The WebSocket client connects to `VITE_BACKEND_WS_URL` (from `.env`); the URL must be configurable via environment variable.
- **FR-4:** WebSocket messages follow a typed discriminated union: `{ type: "token", content: string }` for streamed text and `{ type: "scene_image", url: string }` for scene images.
- **FR-5:** All React components are functional components with TypeScript types; no `any`.
- **FR-6:** The app has at least one Vitest unit test per new custom hook introduced (e.g. `useWebSocket`).
- **FR-7:** Structured logging replaces `console.log`; production paths must not contain raw `console.log` calls.
- **FR-8:** The layout on desktop is a two-panel design: chat panel (left, ~60% width) + scene image panel (right, ~40% width).
- **FR-9:** The WebSocket message discriminated union (`FR-4`) is defined once in `/packages/shared-types` and imported by both `apps/frontend` and the backend service; local copies in individual workspaces are not allowed.

## Non-Goals (Out of Scope)

- Persistence of chat history across page reloads (no backend session save in this iteration).
- NPC portraits, inventory HUD, or R3F canvas integration (Phase 4).
- Voice input / speech-to-text (Phase 5).
- Authentication or multi-user support.
- Mobile-native app; responsive web only.
- Modifying the backend WebSocket protocol beyond what is needed for FR-4 alignment.
